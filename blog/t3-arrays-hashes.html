Arrays vs. Hashes

201502.17

Arrays: The Structured List

Arrays are Ruby's way of collecting objects in a list form. It's sort of like an ordered list of things, such as:

Syllabus:

  1. Git & GitHub
  2. HTML & CSS
  3. Ruby Basics

However in Ruby, an array is written like this:
array_name = [item 1, item 2, item 3, etc]

So the list above could be re-written as:
syllabus = ["Git & GitHub", "HTML & CSS", "Ruby Basics"]

The items (or in Ruby-speak, elements) in the array can be referred to by calling the array name followed by the position (called the index) of the element in the list. For example:
array_name[1]

To call an element from the array named "syllabus" you'd write:
syllabus[1]
Where the number "1" can be any number that references the position (index) of the element in the array.

You could also call the reverse. Let's say you already know the element, but you don't know where it is in the list. To find the index, you'd call the index method:
syllabus.index(2)
Which will return "Ruby Basics".

The only major difference of significance between an array and a normal ordered list is the counting. Typically, you count starting from one, followed by two, then three, four, five and so on. In Ruby, you start at zero. Therefore, instead of "Git & GitHub" being the first item on the list, it is the zeroth element in the array. "HTML & CSS" is the first and "Ruby Basics" is the second.

Going back to the example of how arrays are written in Ruby, we can write the index below each element of the actual position it occupies. Thus,
array_name = [item 1, item 2, item 3, etc]
0,1,2,3

Hashes: The Chaotic Array

Hashes are practically arrays gone crazy. Instead of an ordered list, like a syllabus of topics organized by week, it is more of a random to-do list. It doesn't matter what order anything is in, because instead of using numbers to order the list, it uses what's known as "keys". And the elements of the keys are called "values", together they are known as key-value pairs. You write a hash as follows:
hash_name = { key1 => value1, key2 => value2, key3 => value3, etc. }

An example hash could be something like this:
my_pets = { "Pinkie" => "cat", "Tiger" => "cat", "Missy" => "dog"}

Calling an entry in a hash is just like in an array, you would write the hash name followed by the key of the value you're looking for.
my_pets["Missy"]

And just like arrays, if you know the value but not it's key, you can call the index method:
my_pets.index("Tiger")

NOTE: A key must be unique, otherwise if you were to write the hash above with the key-value pairs switched, the value of the key "cat" would be overridden by the second entry.
my_pets = { "cat" => "Tiger", "cat" => "Pinkie" , "dog" => "Missy"}
Calling the value for "cat" only results in "Pinkie" since it was entered second and essentially replaced the first entry.

It might seem weird to want an unordered list. I mean, isn't the definition of list "a series of numerals" (thanks Merriam-Webster)? What purpose would an unordered list serve? Good question, and now onto...

Array or Hash?

As an ordered list, when to use an array is actually quite simple. If you want a list that always stays in a certain order, you use an array.

For example, lets say you have a sign-up list for a popular class at your local gym. Only the first 25 people to sign up are guaranteed a space in the class. The order of sign ups are important, therefore you would use an array to store this information. As a each person signs up, they are added onto the end of the array. The instructor can then see exactly when each person signed up and let those first 25 people in.

Hashes, on the other hand, are used when you have a list that doesn't need the structure of any particular order. As such, a hash actually may not keep the order you wrote it in. So if you called the my_pets hash above, it could print back to you with the key-value pairs in any order.
my_pets = { "Tiger" => "cat", "Pinkie" => "cat", "Missy" => "dog"}

You could use a hash when you want to store any kind of information (value) that has a certain characteristic (key) associated with it. For example, lets say there's a hot night club you've been dying to get into. One day you just so happen to meet the owner and now you're on the list! Unfortunately for some, that list also includes the "people barred from this club". You could certainly use a hash to store this information. You could store everyone's name as the unique key and a "yes" for entry or a "no" for denied. Using a hash, the bouncer could look up the person's name, and see whether or not to give them entry. The order on the list doesn't matter, it's the person's name (the unique key) and the "yes" or "no" value that's important here.

Conclusion

So the point of all this? Arrays and hashes are used to store a collection of information whether that be an ordered list or not. If order matters, you use an array. If it doesn't matter and what you need is a unique identifier to store some piece of information, you use a hash that has key-value pairs.
